# Image to Game

把用户图片变成可玩的网页小游戏，并支持持续迭代。先判断走哪条工作流：

- **用户给了图、要新游戏** → 工作流一（0-1 创建）；若请求类型与已有模板一致 → 工作流三（模板换皮，更快）
- **已有游戏、要改东西** → 工作流二（1-N 迭代）

## 通用原则（任何情况都遵守）

1. 不用游戏引擎：单个 `index.html` + 素材文件，原生 Canvas + JavaScript。
2. 一切图片素材带程序化回退：加载失败自动改用代码绘制，游戏永不白屏（注入模式见 [references/assets.md](references/assets.md)）。
3. 玩法正确性靠机器人验证，不靠肉眼：`scripts/bot_harness.js` 必须输出 `WIN: true`。
4. 视觉验收靠同步快进截图（`scripts/make_ff_pages.py`），不靠"等几秒再截"。
5. 每轮结束三件事：**交付 → 快照 → 备份**。意图固定，路径与工具随环境替换（见"环境适配"一节）：交付 = 成果放进**持久且用户可访问**的目录（每个项目独立子目录，`index.html` 在根）；快照 = 环境有版本工具就调用；备份 = 源码复制到持久工作区，不能只留在临时目录。
6. **所有游戏竖屏锁定**：游戏区永远是手机竖屏比例，横屏窗口时居中显示 9:16 舞台、两侧留边，不做横屏布局（实现见各类型包）。
7. **2D 有左右移动的游戏必须配跑步序列帧**（每状态 4 帧，一张 2×2 图切出）：生成与切帧见 [references/assets.md](references/assets.md)，播放代码见类型包。
8. **必须有资源加载门**：素材全部 `complete` 前不开局（显示加载进度），加载失败也照常放行（走各自的代码回退），绝不白屏也绝不占位圈开局。UI 不用生图，用 Kenney UI 组件 + 设计令牌，见 [references/ui-kit.md](references/ui-kit.md)。

## 工作流一：0-1 创建（七步）

1. **分析图片与需求**：read_file 看图，定主角、主题映射（收集物/敌人/障碍/终点）。**类型判断（询问门）**：
   - 用户已指明类型（"超级玛丽式""类似 XX""XX 那样的"）→ 不问，直接做；
   - 只说"做个小游戏"没提类型 → 用 ask_user **一次性**问清：游戏类型为主（选项结合图片主题给 2-4 个具体的，如"横版闯关（类似超级玛丽）/ 3D 跑酷（类似地铁跑酷）"，标注推荐项），最多再加一个难度/受众问题；
   - 得到答复前不开工；严禁分多轮连环追问。
2. **素材流水线**：按 [references/assets.md](references/assets.md) —— 主角抠图（`scripts/remove_bg.py`）、提取主色（`scripts/extract_palette.py`）、以原图为风格参考 AI 生成全套素材、清理水印与裁剪（`scripts/clean_sprite.py`）。
3. **写一页 GDD**：按类型包模板填（平台跳跃 → [references/gdd.md](references/gdd.md)，含已验证的物理基线与关卡红线）。
4. **单文件实现**：照 [references/game-patterns.md](references/game-patterns.md) 的结构写（瓦片碰撞、跳跃手感、敌人 AI、视差、合成音效、横竖屏自适应、触控）。
5. **机器人通关验证（硬门槛）**：`node scripts/bot_harness.js <index.html>`，失败先改关卡不改机器人。
6. **截图验收**：`scripts/make_ff_pages.py` 生成快进页 + 无头 Chromium 截图，至少验收标题/中段/终点/竖屏/横屏窗口（确认居中留边）5 张。
7. **交付**：按通用原则第 5 条。

## 工作流二：1-N 迭代（修 bug / 调手感 / 换素材）

按 [references/iteration.md](references/iteration.md)：恢复项目 → 按"症状→病因→修法"对照表定位 → 最小修改 → 机器人+截图回归 → 存新版本。换单个素材只走 assets.md 的单品流程，不动其他代码。

## 工作流三：模板换皮（类型匹配模板时的快速路径）

请求的类型与已有模板一致、且 `templates/` 目录存在时（见类型包一节的获取说明），克隆模板换皮，不从零构建：

1. **复制模板**：`templates/<类型>/` 复制为新项目目录。
2. **换主题素材**（只换文件，不动代码）：按 [references/assets.md](references/assets.md) 流水线，从新图生成全套素材，**文件名与模板保持一致**直接覆盖（素材全部按文件名加载且有代码回退）。
   - 2D 模板：主角 + 跑步帧 ×4（A→B→C→A）、收集物、敌人、终点、障碍、背景、地形砖 ×6
   - 3D 模板：标题图、收集物、天空图；程序化角色改配色/特征参数（runner-3d.md 有清单）
3. **改主题令牌**：游戏名/标题文案、UI 设计令牌（主色/深色，用 `scripts/extract_palette.py` 从新主角取色）；需要时把 Kenney 按钮换成包内其他颜色族（见 [references/ui-kit.md](references/ui-kit.md)）。
4. **不能动**：物理基线、关卡/刷怪红线、加载门、竖屏锁定、机器人——换皮只改"皮"。
5. **验证与交付**：与 0-1 相同硬门槛（机器人通过 + 快进截图目检 5 张）。

## 类型包

- **2D 平台跳跃**（超级玛丽式）：[references/gdd.md](references/gdd.md)（设计基线 + 关卡配方）+ [references/game-patterns.md](references/game-patterns.md)（实现模式），机器人 `scripts/bot_harness.js`。
- **3D 跑酷**（神庙逃亡/地铁跑酷式）：[references/runner-3d.md](references/runner-3d.md)（设计基线 + 程序化建模 + 程序化角色 IP 复刻），机器人 `scripts/runner_bot.js`，引擎 three.js r147 本地打包。
- 新增游戏类型：在 `references/` 下加一个同结构类型包（设计基线 + 实现模式 + 配套机器人脚本），SKILL.md 通用流程不变。每个包必须实战验证过才可写入。

两个已验证类型另有**完整可运行的参考实现**（汪汪主题，可直接换皮）：`templates/platformer-2d/`、`templates/runner-3d/`。模板体积大**不随 .skill 包分发**，与本 skill 同仓库获取。**若 skill 旁没有 templates/ 目录**（单独安装的包），说明无模板可用——直接走工作流一，类型包已含全部构建基线，不依赖模板。

## 环境适配（换 Agent / 换环境时替换这三处）

skill 只有三处依赖具体环境。**意图不变，实现可换**——下表"本环境实现"列只是示例：

| 项 | 不变的意图 | 本环境实现（示例） | 其他环境怎么换 |
|---|---|---|---|
| AI 生图 | 以原图为风格参考、透明底 PNG 产出全套同画风素材 | image_generation 插件（命令见 [references/assets.md](references/assets.md)） | 换成环境自带的生图工具；prompt 策略、比例/透明底约束照搬 |
| 交付目录 | 成果放在持久且用户可访问的位置，每项目独立子目录 | `/mnt/agents/output/<项目名>/` | 项目工作目录 + 起本地 http 服务，把地址给用户 |
| 版本快照 | 每轮结束留一个可回滚、可预览的版本 | `build_version`（type=html） | git commit 打 tag，或直接告知用户打开方式 |

## 常见坑（必须避免）

1. HTML 结尾必须写 `</body></html>`：缺失会导致按 `</body>` 定位的注入静默失效。
2. 临时目录随时可能被清空（跨对话轮次即失效）：代码写完立即备份到持久工作区。
3. 交付与快照的路径必须在持久存储上：放临时目录会让用户拿到 404 / 丢文件。
4. 多个 `<script>` 块共享全局词法作用域：可跨块引用 const/let，但重名声明全崩。
5. 浏览器截图工具传 url 会重新加载页面：动态画面用同步快进，不等时。
6. 坐标系不混用：世界实体用世界坐标（渲染 `- camX`），HUD/界面用画布坐标；视野宽度一律 `VIEW_W` 不写死 `W`。
7. 注入/替换前先断言目标字符串存在（如 `'</body>' in html`），替换静默失败最费调试时间。
8. AI 生成偶发 HTTP 424：sleep 20-30s 重试；2K 不稳就降 1K。
9. shell 后台并行任务不要和 `cd`/`&&` 混用（任务会在错误目录执行）：用绝对路径。
10. AI 透明素材角落常带"AI生成"水印：连通块清理（clean_sprite.py）；不透明背景图直接裁掉底部条带。
11. AI 背景容易画出以假乱真的"假平台"：prompt 必须写 no platforms / no blocks。
12. 类型不明的请求不要闷头按默认类型做（方向错了全部白做）：按询问门一次问清；但也别过度追问——能一轮问完的不分两轮。
13. 验证运动/动画画面时，快进结束后要保持按键输入（如 `keys.ArrowRight = true`）：松键后角色会在截图前减速停下，拍到的永远是静止帧。
14. AI 序列帧一张图出多帧时，先做连通块清理再按格切（顺序反了格子中线会偏），切完必须拼接目检 4 帧。
15. 循环序列帧必须首尾同帧（A→B→C→A）：4 个不同姿势循环会在回绕时跳变；末帧用代码复制首帧，别让 AI 画"一样的两张"。
16. 手机网络下 10MB 级素材没有加载门，玩家会看到占位圆圈/素模——桌面本地截图永远测不出（本地秒载），必须从机制上保证"素材就绪才开局"。
