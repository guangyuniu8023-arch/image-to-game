# 新类型 0-1 设计推演（没有类型包时，先设计、后实现）

适用：请求的游戏类型不在已有类型包（2D 平台跳跃 / 3D 跑酷 / 消消乐 / 竖版弹跳）里。通用流程与四级验证不变；本文件定义的是**设计推演策略，不是 GDD 模板**：模块用于防漏设计决策，不规定标题、表格、篇幅或书写顺序。简单游戏可以合并模块；不适用项写明原因，不得无声跳过。铁律不变：**实战验证 → 用户验收 → 蒸馏成新类型包**。

## GDD、assets.md、ui-kit.md 的职责边界

- **GDD 负责“设计什么、为什么”**：玩法规则、角色职责、素材职责、Sprite 状态、HUD 信息与验收口径都在生产前闭合。
- **[assets.md](assets.md) 负责“怎样生产素材”**：prompt、视角/尺寸执行、抠图、切帧、清理、回退和一致性验收；它不能替代素材设计决策。
- **[ui-kit.md](ui-kit.md) 负责“怎样生产和集成界面”**：组件底材、代码叠字、状态、九宫格、对齐与截图验收；它不能替代 HUD 信息架构。

设计顺序固定为：**图片/需求分析 → GDD 八模块决策 → 一致性检查 → 素材生产 → 实现**。禁止先批量生成素材，再用已经生成的图片倒逼玩法和 HUD。

## 八个设计模块（产出 = 可执行的设计决策集合）

### 1. 核心体验、循环与输入

- 输入：用户需求、游戏类型、图片特征表、平台/时长等明确约束。
- 一句话写清玩家扮演谁、每 10 秒做什么、承受什么压力、如何获得进展。
- 定义核心循环、主要动作、WIN/LOSE 和单局目标；声明输入维度（离散方向+瞬时动作 / 单轴连续量 / 点选拖拽，或组合），按 [controls.md](controls.md) 落到语义输入。
- 输出：核心循环、玩家动作、输入模式、胜负条件；不是宣传语。

### 2. 机制、参数与可行性红线

- 输入：模块 1 的核心循环/动作/胜负条件、同类规则知识或已验证类型包。
- 从经典同类游戏或已验证类型包取机制参数基线，写明来源与理由，不允许凭空编数。成熟规则可用内部知识但须标“未联网核实”；类型陌生或无把握时再联网核实。
- 影响可玩性的硬数字必须由公式、模拟或机器人实证收口，不能以记忆或搜索结果直接背书。
- 每个新类型至少推 3 条红线，格式为“风险 → 算术/不变量 → 代码约束 → 验证证据”。例如跳跃高度反推台阶上限，或棋盘必须保证无现成消除且至少一步可走。
- 输出：规则、参数基线、来源、红线和调参牵连项。

### 3. 关卡、内容与进度

- 输入：模块 1 的玩家动作与目标、模块 2 的机制能力边界和红线。
- 有关卡型给分段配方和难度曲线；每段只引入一个新元素，组合放后半程。
- 无关卡型改为参数进度：步数/时间、目标、障碍梯度、机制解锁或生成密度。
- 每个敌人、障碍、道具必须说明它要求玩家做出什么不同决策；只增加数量、不改变决策的内容应合并或删除。
- 输出：一局的教学—强化—组合—高潮结构，以及供 bot 检查的内容完整性条件。

### 4. 角色职责、主题与视觉方向

- 输入：原图特征表、模块 1 的玩家身份/动作、模块 3 的玩法实体与内容结构。
- 复用 [assets.md](assets.md) 的角色特征提取表，声明角色融合档位：A 操作主体 / B 主体驱动者 / C 看板娘。角色必须承担操作、驱动或解说职责，不能只贴在场景里。
- 从标志性元素、身份与世界观线索推导主题域；收集物、敌人、障碍、终点和背景逐项说明“为什么属于这个角色的世界”。
- 产出视觉设计简报：画风、描边、主色/点缀色、形状语义、对比与密度方向。具体判据和执行方法引用 assets.md，不在 GDD 复制生产教程。
- 输出：角色职责、主题映射与视觉简报。

### 5. 素材系统设计

- 输入：模块 1~4 的玩法实体、反馈事件、关卡内容、角色职责和视觉简报。
- 从玩法实体、反馈事件和界面信息反推素材，不从平台跳跃固定清单照抄。
- 每项素材至少闭合：**玩法/信息职责、状态或变体、视角、游戏内尺寸锚点、绘制层与锚点、与碰撞盒关系、AI/代码/混合生产方式、加载失败回退**。
- 背景和装饰也要声明职责与密度边界，禁止出现会被误认成平台、障碍、按钮的假交互元素。
- 输出：可直接交给 assets.md 的素材生产清单；此阶段只定设计，不生成文件和 prompt。

### 6. Sprite 与动画状态设计

- 输入：模块 1 的玩家动作、模块 4 的角色职责、模块 5 的角色素材与锚点设计。
- 从玩家动作、角色职责和状态机反推动画状态，不把“生成跑步四帧”当成 Sprite 设计本身。
- 每个状态说明：触发事件、静态图/序列帧/程序化动画、帧数与顺序、帧率、循环方式、退出条件、朝向、脚底/手部等锚点。
- A 操作主体按实际动作设计 idle/move/action/hurt/win/lose；2D 左右移动的 move 状态遵守 assets.md 的 4 帧 A→B→C→A 红线。B 主体驱动者优先设计发球/装填/受击等事件帧；C 看板娘设计事件联动表，不强造跑步帧。
- 输出：Sprite/动画状态计划，随后由 assets.md 负责生图、清理、切帧和拼接目检。

### 7. HUD/UI 信息与视觉设计

- 输入：核心循环中的决策变量、胜负/资源状态、竖屏布局约束、模块 4 的视觉简报。
- 先决定“玩家当前做决策必须知道什么”，再决定显示方式；区分常驻信息、情境信息和角落操作，避免把所有功能都做成大按钮。
- 明确信息优先级、屏幕分区、组件及状态、主题 icon、标题/结算结构；从角色画风和游戏节奏推导 ui-kit.md 的线条、质感、形状、密度四属性。
- GDD 记录信息架构与视觉方向；UI 底材 prompt、九宫格、文字叠加、按下/禁用态和对齐验收交给 ui-kit.md 执行。
- 输出：HUD 信息架构、布局方向、组件状态清单与 UI 四属性。

### 8. 架构、验收与一致性检查

- 输入：模块 1~7 的全部设计决策。
- 定义状态机（loading/title/play/win/lose）、主循环骨架、数据表示（瓦片网格/棋盘数组/实体表）、机器人怎么玩、WIN 判定和死局处理。
- 生产素材前逐项检查：每个玩法实体是否有视觉表达和回退；每个角色状态是否有 Sprite/程序化反馈；每个关键数值是否进入 HUD 或无需显示；素材尺寸是否锚定游戏单位和碰撞盒；HUD 是否遮挡玩法；是否存在没有玩法/信息职责的多余素材。
- 输出：实现架构、机器人验收口径、玩法—素材—Sprite—HUD 一致性结论。任何一项不闭合，退回对应模块，不进入素材生产。

## 推演之后

- **生产**：把已通过一致性检查的视觉简报、素材清单和 Sprite 计划交给 assets.md；把 HUD 信息架构和 UI 四属性交给 ui-kit.md。reference 只能执行或验收既有决策，发现冲突时返回 GDD 修正，不得现场另起一套设计。
- **实现**：通用骨架（加载门/回退注入/竖屏锁定/触屏输入/合成音效）+ 本类型机制；正式逻辑消费 GDD 的语义输入、素材锚点、Sprite 状态和 HUD 信息。
- **验证**：四级验证不变；机器人按类型适配（参照 bot_harness.js 的 DOM-shim 思路），截图同时验证玩法场景、Sprite 状态、HUD 信息层级和素材一致性。
- **蒸馏**：用户验收后，把八模块中的已验证结论固化成 `references/<类型>.md` 类型包（设计基线 + 视觉/素材基线 + 实现模式 + 配套机器人脚本），有完整参考实现时附 `templates/<类型>/`。

## 双层蒸馏约定（任何经验进 skill 的强制结构）

经验与最佳实践必须**成对**沉淀，缺一不可：

- **策略栏（指导设计）**：为什么这么设计——推导思路、红线算术、来源标注（实战案例 + 实测数据）。没有来源的"应该"禁止写入。
- **执行栏（指导执行）**：怎么落地、怎么验证——可照做的步骤/命令/参数。**优先沉淀为可跑载体**（`scripts/` 工具、`templates/` 骨架、bot 断言），文字步骤只是兜底；文字靠执行 agent 临场重写代码，每次都会走样（v5 教训：钉基线测量写了文字仍被做错，做成 scripts/audit_ui_align.py 才闭环）。
- 对照检查：只有策略没有执行 = 空谈；只有执行没有策略 = 死表（换个需求就失效）。
- 适用范围：新类型包、教训/陷阱条目、铁律、验收方法、变体条目——全部按此两栏书写。

## 变体蒸馏（模板改造 → 已验证变体）

**变体 = 核心循环不变、基于已有类型/模板的改造**。四类都算变体：

| 类别 | 改的是什么 | 例子 |
|---|---|---|
| UI/布局 | HUD 布局、标题/结算版式、控件排布 | 比分从 chip 改中央大数字、竖屏 HUD 重排 |
| 机制 | 能力/规则改动 | 二段跳、加速跑、受伤规则变化 |
| 内容 | 基于当前玩法新增元素 | 消消乐+冰块障碍、平台跳跃+新敌型/新道具 |
| 规则/玩法 | 胜负条件、节奏制式改动 | 限步↔限时、目标分↔任务收集、通关制↔无尽制 |

**红线影响深度递增**：UI 变体只动表现（脚本验收即可）；机制/内容变体动参数（红线算术重推）；**规则/玩法变体动胜负判定——结算时机、奖励公式、bot 验收口径三处必须同步改，漏一处 bot 就误报**（限时制首案实证）。

**登记处**：所属类型包尾部"已验证变体"段。每条四栏（双层约定的应用）：

1. **名称 + 类别**（UI/机制/内容/规则）；
2. **delta**：相对基线改了什么——只记 delta，基线仍是唯一事实源，**禁止整段复制基线**（两处真相必漂移）；
3. **红线重推**：受影响的判定链/红线重新算术（机制/内容/规则变体必做：参数改动可能推翻原红线，如二段跳 → 台阶/缺口上限）；**UI 变体改跑 `audit_ui_align.py` + `make_contact_sheet.py`**；
4. **验证证据**：机器人回归结果 + 截图/脚本验收结论 + 用户验收日期。未验证不入库（铁律不变：实战验证 → 用户验收 → 蒸馏）。
5. **对比可视化交付（用户制度）**：每次验证后必须给用户**前后对比**，不允许只报数值结论——数值证明"过了"，对比让用户判断"好不好"。**对比形态按变体类别分级，选错形态等于没证据**：
   - **UI/布局变体** → 同区域截图上下拼贴（基线 vs 变体），辅以像素测量数据；
   - **机制/内容/规则变体** → **可玩 A/B 链接**（基线与变体各自完整可玩，最好做一页 A/B 对比导航页，写清两版的判定/奖励/节奏差异与验收关注点）——玩法差别只有亲手玩才能验收，**HUD 截图对玩法不构成证据，只能当附图**。

**边界规则**：核心循环改变（玩家每 10 秒做的事变了）→ 不是变体，走八模块推演成独立类型包；换皮（只换素材/令牌、不动机制）不是变体，不登记。

## 自进化三层（变体积累 → 通用规则 → 重组）

变体条目只增不删会变成新的死表——**案例积累必须配泛化机制**，skill 才能"越用越聪明"。

### 触发条件（什么时候自迭代，精确口径）

- **L1 案例入库**：每次实战验证 + 用户验收后执行，追加到类型包"已验证变体"段。常态动作，无需额外批准（验收本身就是批准）。
- **L2 泛化**：两个条件**同时满足**才可发起——
  - ① **数量/命中**：同一类型包、同一类别的已验证变体 ≥3 条（计数只算已入库条目；修 bug/对齐修正属铁律/陷阱条目，**不算变体、不计数**）；或用户再次命中相似需求；
  - ② **用户批准**：①满足只产生"提案权"——把《泛化提案》（候选案例清单 / 提取的共性规则 / 适用范围 / 对旧条目的处置）发给用户，**批准后才写入**。agent 不得自主泛化。
  - 示例：限步/限时/任务制三条规则类变体 → 泛化提案《制式变体通用规则：改胜负判定必须同步改结算时机/奖励公式/bot 口径三处》。
- **L3 重组**：**只能由用户发起**（用户说"整理一下"，或用户批准一份重组提案）。发现规则冲突时 agent 可以提案，但任何"修改/移动/废弃已有条目"的动作都属 L3，禁止自主执行。

### 防误触发纪律（不能被条件句带跑）

1. **触发 ≠ 执行（总闸门）**：L2/L3 的条件满足永远只给"提案权"，写入一律用户批准——与"实战验证 → 用户验收 → 蒸馏"铁律同源。没有任何条件能让 agent 自主改写 skill 正文规则。
2. **计数口径**：以类型包"已验证变体"段的条目数为准，跨类型包不合并计数；项目级改造未走完入库流程的不算。
3. **只加不改边界**：L1/L2 只能**追加**新条目。执行中发现自己想"顺手优化/重写/合并"已有条目时——停止，那属于 L3，先提案。
4. **变体与修复的边界**：修 bug、对齐修正、API 陷阱是对应文档的铁律/陷阱条目（ui-kit.md 等），不进变体段、不进 L2 计数——防止用修复记录凑数提前触发泛化。
5. **推送门禁**：任何蒸馏推送前 `bash scripts/selfcheck.sh` 必跑（verification.md 第 0 级），全绿才许推。

### 四道闸门（防泛化丢精华，L3 是有损压缩、最危险环节）

1. **精华判定显式化**：按溯源数据（命中记录、验收日期、实测数据）排序，不靠"觉得重要"；
2. **只做加法不做删除**：案例永不删；旧泛化被取代时移入"历史"段标注取代关系，不物理删除；泛化永远保留案例指针，证据链不断；
3. **覆盖性 diff 检查**：重组前后逐条回答"旧规则覆盖的每个坑，新规则还覆盖吗"——新覆盖面 ⊇ 旧覆盖面才允许替换，缺一条补一条；
4. **用户验收 + 实战复检**：重组 diff（保留/合并/废弃及理由）给用户过目才入库；泛化规则首次实战使用时跟踪复检——同类坑还踩说明泛化丢了东西，当场补回，回路自愈。
