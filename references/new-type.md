# 新类型 0-1 设计推演（没有类型包时，先设计、后实现）

适用：请求的游戏类型不在已有类型包（2D 平台跳跃 / 3D 跑酷 / 消消乐 / 竖版弹跳）里。通用流程与四级验证不变；本文件补的是"根据图和需求，产出游戏架构/玩法/关卡设计详细说明"的方法。铁律不变：**实战验证 → 用户验收 → 蒸馏成新类型包**。

## 五步推演（产出 = 一页 GDD 详细说明）

1. **核心循环一句话**：玩家每 10 秒在做什么？（如"交换棋子制造连线并追求连锁"、"向右跑跳躲障碍到达终点"）——它决定主循环结构、输入方式和胜负条件。**同时声明输入维度**（离散方向+瞬时动作 / 单轴连续量 / 点选拖拽，或其组合），按 [controls.md](controls.md) 输入模式推导落到模式，不许为类型临时发明方案。
2. **机制参数基线**：从经典同类游戏取基线，**写明来源与理由**，不允许凭空编数。（例：消消乐 8×8/6 色/30 步/目标 3000，参照开心消消乐经典关卡；平台跳跃 GRAV 2400/JUMP -860，来自已验证的汪汪基线。）
   **基线考据策略**：经典同类规则优先取模型内部知识并如实标注（如"来源：Doodle Jump 规则框架，内部知识，未联网核实"）；仅当类型完全陌生、内部知识无把握时才联网核实。无论来源为何，**影响可玩性的硬数值一律以红线推导 + 机器人实证背书，不以记忆或搜索结果直接背书**。
3. **可行性红线推导法**（承自平台跳跃）：从机制参数用算术推导设计约束，把"可通关/可玩"变成可检查的数学条件，**每个新类型至少推 3 条**。
   - 范例（平台跳跃）：跳跃最大高度 vy²/2g ≈154px → 台阶高差 ≤3 格；跳跃远距 ≈236px → 缺口 ≤4 格。
   - 范例（消消乐）：实测平均单步得分 ≈150-200 → 目标分 ≤ 步数×均值×0.7；机器人上限的 1/3 以内做普通玩家难度带；死盘必现 → 洗牌必须保证"无现成消除 + 至少一步可走"。
4. **关卡/进度结构**：有关卡型给分段配方 + 难度曲线（每段只引入一个新元素，组合放后半程——承自平台跳跃配方）；无关卡型（益智/消除）改为参数进度（步数/目标分/障碍梯度/解锁机制）。
5. **架构与验收定义**：状态机划分（loading/title/play/win/lose）+ 主循环骨架（参照 game-patterns.md 的通用结构）+ 数据表示（瓦片网格/棋盘数组/实体表）；**机器人怎么玩、赢的判定、死局处理——写清楚了才许写代码**。

## 推演之后

- **实现**：通用骨架（加载门/回退注入/竖屏锁定/触屏输入/合成音效）+ 本类型机制；触屏按 [controls.md](controls.md) 两层走（语义输入 → 按第 1 步声明的输入维度落模式，触屏默认无设备检测）；素材走 assets.md 全套策略（角色图驱动 → 主题推导 → 设计规范 → 流水线）；UI 按 ui-kit.md 主题适配策略（骨架固定、皮肤推导）。
- **验证**：四级验证不变；机器人按类型适配（参照 bot_harness.js 的 DOM-shim 思路，如消消乐的合法步模拟 + 死局断言）。
- **蒸馏**：用户验收后，把五步推演的结果固化成 `references/<类型>.md` 类型包（设计基线 + 实现模式 + 配套机器人脚本），有完整参考实现时附 `templates/<类型>/`。

## 双层蒸馏约定（任何经验进 skill 的强制结构）

经验与最佳实践必须**成对**沉淀，缺一不可：

- **策略栏（指导设计）**：为什么这么设计——推导思路、红线算术、来源标注（实战案例 + 实测数据）。没有来源的"应该"禁止写入。
- **执行栏（指导执行）**：怎么落地、怎么验证——可照做的步骤/命令/参数。**优先沉淀为可跑载体**（`scripts/` 工具、`templates/` 骨架、bot 断言），文字步骤只是兜底；文字靠执行 agent 临场重写代码，每次都会走样（v5 教训：钉基线测量写了文字仍被做错，做成 scripts/audit_ui_align.py 才闭环）。
- 对照检查：只有策略没有执行 = 空谈；只有执行没有策略 = 死表（换个需求就失效）。
- 适用范围：新类型包、教训/陷阱条目、铁律、验收方法、变体条目——全部按此两栏书写。

## 变体蒸馏（模板改造 → 已验证变体）

**变体 = 核心循环不变、基于已有类型/模板的改造**。四类都算变体：

| 类别 | 改的是什么 | 例子 |
|---|---|---|
| UI/布局 | HUD 布局、标题/结算版式、控件排布 | 比分从 chip 改中央大数字、竖屏 HUD 重排 |
| 机制 | 能力/规则改动 | 二段跳、加速跑、受伤规则变化 |
| 内容 | 基于当前玩法新增元素 | 消消乐+冰块障碍、平台跳跃+新敌型/新道具 |
| 规则/玩法 | 胜负条件、节奏制式改动 | 限步↔限时、目标分↔任务收集、通关制↔无尽制 |

**红线影响深度递增**：UI 变体只动表现（脚本验收即可）；机制/内容变体动参数（红线算术重推）；**规则/玩法变体动胜负判定——结算时机、奖励公式、bot 验收口径三处必须同步改，漏一处 bot 就误报**（限时制首案实证）。

**登记处**：所属类型包尾部"已验证变体"段。每条四栏（双层约定的应用）：

1. **名称 + 类别**（UI/机制/内容/规则）；
2. **delta**：相对基线改了什么——只记 delta，基线仍是唯一事实源，**禁止整段复制基线**（两处真相必漂移）；
3. **红线重推**：受影响的判定链/红线重新算术（机制/内容/规则变体必做：参数改动可能推翻原红线，如二段跳 → 台阶/缺口上限）；**UI 变体改跑 `audit_ui_align.py` + `make_contact_sheet.py`**；
4. **验证证据**：机器人回归结果 + 截图/脚本验收结论 + 用户验收日期。未验证不入库（铁律不变：实战验证 → 用户验收 → 蒸馏）。
5. **对比可视化交付（用户制度）**：每次验证后必须给用户**前后对比**，不允许只报数值结论——数值证明"过了"，对比让用户判断"好不好"。**对比形态按变体类别分级，选错形态等于没证据**：
   - **UI/布局变体** → 同区域截图上下拼贴（基线 vs 变体），辅以像素测量数据；
   - **机制/内容/规则变体** → **可玩 A/B 链接**（基线与变体各自完整可玩，最好做一页 A/B 对比导航页，写清两版的判定/奖励/节奏差异与验收关注点）——玩法差别只有亲手玩才能验收，**HUD 截图对玩法不构成证据，只能当附图**。

**边界规则**：核心循环改变（玩家每 10 秒做的事变了）→ 不是变体，走五步推演成独立类型包；换皮（只换素材/令牌、不动机制）不是变体，不登记。

## 自进化三层（变体积累 → 通用规则 → 重组）

变体条目只增不删会变成新的死表——**案例积累必须配泛化机制**，skill 才能"越用越聪明"：

- **L1 案例层**：变体条目（上文四栏格式）。常态动作，每次实战后入库。
- **L2 泛化层**：同类变体 ≥3 条、或用户再次命中相似需求时触发——提取共性合并成通用规则，挂到类型包或策略文档正文（例：限步/限时/任务制三条 →《制式变体通用规则：改胜负判定必须同步改三处》）。泛化条目同样过双层约定（必须带"为什么"），保留到案例的指针，写清适用范围。
- **L3 重组层**：泛化规则之间的再合并。**稀有事件**——只在规则冲突或用户发起整理时做；越往上越少动。

**四道闸门（防泛化丢精华，L3 是有损压缩、最危险环节）**：

1. **精华判定显式化**：按溯源数据（命中记录、验收日期、实测数据）排序，不靠"觉得重要"；
2. **只做加法不做删除**：案例永不删；旧泛化被取代时移入"历史"段标注取代关系，不物理删除；泛化永远保留案例指针，证据链不断；
3. **覆盖性 diff 检查**：重组前后逐条回答"旧规则覆盖的每个坑，新规则还覆盖吗"——新覆盖面 ⊇ 旧覆盖面才允许替换，缺一条补一条；
4. **用户验收 + 实战复检**：重组 diff（保留/合并/废弃及理由）给用户过目才入库；泛化规则首次实战使用时跟踪复检——同类坑还踩说明泛化丢了东西，当场补回，回路自愈。
