# 新类型 0-1 设计推演（没有类型包时，先设计、后实现）

适用：请求的游戏类型不在已有类型包（2D 平台跳跃 / 3D 跑酷 / 消消乐 / 竖版弹跳）里。通用流程与四级验证不变；本文件补的是"根据图和需求，产出游戏架构/玩法/关卡设计详细说明"的方法。铁律不变：**实战验证 → 用户验收 → 蒸馏成新类型包**。

## 五步推演（产出 = 一页 GDD 详细说明）

1. **核心循环一句话**：玩家每 10 秒在做什么？（如"交换棋子制造连线并追求连锁"、"向右跑跳躲障碍到达终点"）——它决定主循环结构、输入方式和胜负条件。**同时声明输入维度**（离散方向+瞬时动作 / 单轴连续量 / 点选拖拽，或其组合），按 [controls.md](controls.md) 输入模式推导落到模式，不许为类型临时发明方案。
2. **机制参数基线**：从经典同类游戏取基线，**写明来源与理由**，不允许凭空编数。（例：消消乐 8×8/6 色/30 步/目标 3000，参照开心消消乐经典关卡；平台跳跃 GRAV 2400/JUMP -860，来自已验证的汪汪基线。）
   **基线考据策略**：经典同类规则优先取模型内部知识并如实标注（如"来源：Doodle Jump 规则框架，内部知识，未联网核实"）；仅当类型完全陌生、内部知识无把握时才联网核实。无论来源为何，**影响可玩性的硬数值一律以红线推导 + 机器人实证背书，不以记忆或搜索结果直接背书**。
3. **可行性红线推导法**（承自平台跳跃）：从机制参数用算术推导设计约束，把"可通关/可玩"变成可检查的数学条件，**每个新类型至少推 3 条**。
   - 范例（平台跳跃）：跳跃最大高度 vy²/2g ≈154px → 台阶高差 ≤3 格；跳跃远距 ≈236px → 缺口 ≤4 格。
   - 范例（消消乐）：实测平均单步得分 ≈150-200 → 目标分 ≤ 步数×均值×0.7；机器人上限的 1/3 以内做普通玩家难度带；死盘必现 → 洗牌必须保证"无现成消除 + 至少一步可走"。
4. **关卡/进度结构**：有关卡型给分段配方 + 难度曲线（每段只引入一个新元素，组合放后半程——承自平台跳跃配方）；无关卡型（益智/消除）改为参数进度（步数/目标分/障碍梯度/解锁机制）。
5. **架构与验收定义**：状态机划分（loading/title/play/win/lose）+ 主循环骨架（参照 game-patterns.md 的通用结构）+ 数据表示（瓦片网格/棋盘数组/实体表）；**机器人怎么玩、赢的判定、死局处理——写清楚了才许写代码**。

## 推演之后

- **实现**：通用骨架（加载门/回退注入/竖屏锁定/触屏输入/合成音效）+ 本类型机制；触屏按 [controls.md](controls.md) 两层走（语义输入 → 按第 1 步声明的输入维度落模式，触屏默认无设备检测）；素材走 assets.md 全套策略（角色图驱动 → 主题推导 → 设计规范 → 流水线）；UI 按 ui-kit.md 主题适配策略（骨架固定、皮肤推导）。
- **验证**：四级验证不变；机器人按类型适配（参照 bot_harness.js 的 DOM-shim 思路，如消消乐的合法步模拟 + 死局断言）。
- **蒸馏**：用户验收后，把五步推演的结果固化成 `references/<类型>.md` 类型包（设计基线 + 实现模式 + 配套机器人脚本），有完整参考实现时附 `templates/<类型>/`。
