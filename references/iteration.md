# 1-N 迭代工作流

适用于修 bug、调手感、换素材、换主题、改 UI 和新增功能。项目 `GDD.md` 是设计事实源，`ASSET_LEDGER.md` 是素材生产事实源；类型包只提供基线。

## 六步流程

1. **恢复项目**：取回 `index.html`、素材、GDD、ledger 和上轮验证证据；需要回退时恢复明确版本。
2. **复现定位**：用项目 GDD 模块 8 的固定种子、测试状态、机器人或截图入口复现，不先猜修法。
3. **判断影响模块**：标出八模块中受影响项及牵连项。纯代码 bug 可不改设计；规则、参数、角色职责、素材状态、HUD 信息或验收口径变化必须先更新 GDD。
4. **最小修改**：只改受影响代码/素材。换素材同步更新 `ASSET_LEDGER.md`；生产中出现新设计决策，返回 GDD 重新过一致性门。
5. **类型化回归**：运行类型注册机器人，重拍受影响状态和通用竖横屏状态；规则变化同步检查 WIN/LOSE、奖励公式和机器人断言。
6. **存新版本**：交付 → 快照 → 备份，并记录修改影响的 GDD 模块、验证证据和未解决问题。

## 影响判断

| 用户表述 | 优先检查 | 必须同步 |
|---|---|---|
| “不好玩/太难/太简单” | 模块 1~3 | 参数红线、进度、机器人断言 |
| “角色像贴上去的” | 模块 4、6 | 角色职责、形态、状态反馈、素材清单 |
| “素材不统一/看不懂” | 模块 4、5 | 视觉简报、职责、尺寸/锚点、ledger |
| “白名单动作不可读 / 状态信息确有缺口” | 模块 6 | 动画决策表、生产白名单、状态机、Sprite Contract、Seed/序列帧、root/socket、测试状态 |
| “HUD 挡住/信息乱” | 模块 7 | 信息架构、布局、组件状态、截图 |
| “通关但机器人失败” | 模块 2、3、8 | 规则定义、完整性断言、测试接口 |

## 调参数纪律

- 不跨类型套数值；回到对应类型包或项目 GDD 的公式/不变量。
- 改一个基础参数时列出牵连项，例如速度会影响反应窗，跳跃会影响台阶与缺口，步数/时间会影响目标分。
- 先重推红线，再改代码，最后跑机器人和边界测试。
- 不通过降低断言、修改机器人作弊或无限延长时长来“修复”游戏。

## 换素材纪律

- 单件替换也要核对玩法职责、状态、视角、游戏尺寸、锚点、碰撞关系、生产方式和回退。
- 重新生成后更新 `ASSET_LEDGER.md` 原记录；旧 prompt/配方可保留为历史，但当前版本必须唯一明确。
- 换主角时重新检查角色职责、体型/碰撞、Sprite 状态、HUD 遮挡和世界观映射，不能只覆盖文件。
- 换主角身份、核心动作或视角时，重新产出游戏内 Seed Frame 和动作节拍预览；若用户明确要求先确认美术方向则暂停，否则直接按 GDD 的必要动作生成整条序列帧并在候选中供验收。只换同一身份的色变体可复用当前 Seed。
- 序列帧入引擎前运行 `audit_sprite_frames.py`，并保存带 root/socket 标记的 preview；生产白名单中的 M 动作还要重拍动作时间线，不能只拍一张静态截图。新增动作先回到 GDD 动画决策表过四道门，不在迭代现场直接补状态。
- 全套换主题属于差异 GDD：继承玩法基线，重新闭合模块 4~7，再进入生产。
