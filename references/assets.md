# 素材流水线：让全套素材"从图片里长出来"

目标：用户给什么图，游戏素材就是什么主题、什么画风。按职责选择 AI 生图、Canvas 程序化或混合生产，并以原图特征统一风格。

## 与 GDD 的输入/输出边界

本文件分两种调用模式，不能混用：

- **阶段 A：GDD 设计参考**。输入用户图片事实、玩法实体和角色动作；输出角色职责候选、主题映射判据、视觉简报判据、素材职责/尺寸/锚点和 Sprite 状态设计依据。此阶段不生成正式文件，不写最终 prompt。
- **阶段 B：素材生产与验收**。只有项目 `GDD.md` 已按 [gdd-strategy.md](gdd-strategy.md) 通过一致性门后才能进入。输入已批准的视觉简报、素材生产清单、动画决策表、生产白名单和 Sprite Contract；新角色/新类型只预览白名单内的 Seed Frame 与动作节拍，等用户批准后才继续。最终输出素材文件、归一化 Sprite 帧、root/socket 元数据、回退、preview/contact sheet 和项目 `ASSET_LEDGER.md` 对账记录。

若阶段 B 发现缺少状态、尺寸与碰撞冲突、HUD 遮挡或素材没有职责，退回 GDD 修正并重新过门；禁止在生产阶段静默新增设计。

## 阶段 A：GDD 设计参考

## 角色图驱动策略：让素材"是这个角色的世界"

目标：玩家一眼觉得全套素材与角色图是"一家的"，而不是通用素材库。四步：

1. **角色特征提取表**（看角色图必填，结论写进 GDD）：
   - 形态类别：两足竖高 / 四足横宽 / 其他 → 结合类型包与下文“素材设计规范”决定尺寸基线
   - 画风：赛璐璐动漫 / kawaii 绘本 / 像素 / 扁平矢量 / 厚涂写实 → 决定风格锚点写法（见下表）
   - **描边：粗 / 细 / 无** → UI 线条属性推导直接消费（ui-kit.md ①），不填就得重新看图
   - 主色 + 点缀色：`extract_palette.py` 取色 → UI 令牌与全套素材配色倾向
   - **标志性元素**：配饰/道具/图案（心形腰包、耳机、武器、宠物）→ 收集物与终点建筑的定制线索
   - 性格/世界观暗示：服装风格、年代感、身份 → 敌人/障碍/背景的主题线索
2. **主题定制**：按“主题推导策略”归纳主题域、填充玩法槽位；每个槽位在 GDD 写明选择理由，阶段 A 只产出生产简报，不写最终 prompt。
3. **生产方式决策**：按精度、质感、状态一致性和回退要求，为每项素材选择 AI、Canvas 或混合方式。
4. **生产后验收**：阶段 B 将全套素材与角色图并排做 contact sheet，检查描边、上色、明暗和配色家族；单件跑偏只重做该件。

**风格锚点速查**（替换"Prompt 模式"的默认锚点，全套统一用一个）：

| 画风 | 锚点写法 |
|---|---|
| 赛璐璐动漫 | Japanese anime cel-shading, clean line art, vibrant flat colors |
| kawaii 绘本（默认） | kawaii children's book illustration style, thick dark brown outline, soft flat colors |
| 像素 | 16-bit pixel art, limited palette, crisp pixels, no anti-aliasing |
| 扁平矢量 | flat vector illustration, geometric shapes, bold solid colors |
| 厚涂/写实 | painterly concept art, soft brush strokes, cinematic lighting |

## 角色融合档位：角色在游戏里扮演什么（类型推演时一并声明，写进 GDD）

角色不是默认"站进去"——按玩法主体分三档（打砖块"少女贴挡板当摆设"被用户驳回后沉淀）：

| 档 | 定位 | 适用类型 | 配套规范 |
|---|---|---|---|
| **A 操作主体** | 角色本人被操控 | 平台跳跃/跑酷/弹跳 | 身体直接完成且通过四道门的 M 动作才使用序列帧；其余状态不从类型名自动补齐 |
| **B 主体驱动者** | 角色是玩法物的直接驱动者 | 弹珠/战机/炮塔等驱动类 | 只生产需要身体轮廓或 socket/碰撞/事件帧的白名单动作；角色区不挡主球路/判定点，禁止静态立绘 + 独立玩法控件冒充驱动动作 |
| **C 看板娘** | 与玩法无直接操控关系 | 消除/益智/信息辅助类 | 默认复用已有姿态、台词、UI 或程序化反馈；只有通过四道门的必要 S 动作才进生产白名单 |

**铁律：角色必须有"职责"——操作、驱动、解说三选一**，不许出现"无任何动作职责、纯贴图"的摆设位。

**融合 = 形态与动作重设计，不是贴图**：按玩法视角把角色改造成可操控素材——贴底半身（接球/发球类）、正面/侧面全身（跑跳类）、Q 版（扁平世界需要时）。新形态先作为**游戏内 Seed Frame**放入实际构图，同时检查角色区占比、球路让位、尺寸和判定点；透明图单独好看不算通过。原图只提供 IP 特征（发色/标志元素/配色），**禁止原封立绘贴进场景或只在上面叠一个玩法控件**。

## 背景三层方案（背景不是默认 AI 大图）

背景按三层组装，前两层代码绘制、第三层可选 AI：
1. **底层**：令牌色渐变（代码，深→浅 2~3 stop，永不抢戏、永远无缝）；
2. **中层**：程序化氛围元素（按风格族选形：玻璃/梦幻族 = 星点+光斑+薄雾；卡通族 = 云/山剪影），视差滚动 0.2~0.5，间距取模保无缝（平台跳跃实例见 platformer-patterns.md 视差节）；
3. **图层（可选）**：氛围大图才让 AI 生——必须同时满足：构图合格（见下）+ 亮度锚 + mirror-wrap/crossfade 集成。

**滚动背景无缝红线**（shmup / doodle jump 实踩）：
- **prompt 侧**：滚动类背景构图必须**上下均匀、无地平线、无大焦点元素**——大云/纸飞机/斜挂银河都是"地标"，每块砖重复出现一眼假（mirror-wrap 只能保证像素连续，藏不住结构重复）；"seamless/tileable"写进 prompt 也没用，AI 出不了真无缝——**无缝是集成端的责任，prompt 管均匀**；
- **集成侧**：mirror-wrap（相邻块垂直/水平镜像）+ cover 适配（保持宽高比居中裁切铺满，禁止拉伸变形）；
- **验收侧**：verification.md 第 3 级"滚动验缝"必过；
- 静态背景构图可以分区（如顶部压暗衬砖墙区/UI），但同样过亮度锚。

**亮度锚**（AI 倾向偏亮，实踩两次）：深色族 prompt 显式堆 `very dark / dim / low brightness`（中文：非常暗、低亮度）；**拿到图先测不透明区亮度**（深色族目标 <60，砖墙/UI 区更暗），过亮重新生成，不许调透明度凑合。

## 视觉设计简报（行业法则层：主题定"谁的世界"，这层管"好不好看"）

生成任何素材前先出这份简报（写进 GDD）。**每条规则 = 意图 + 可机检判据；写不出判据的规则不进简报**（防空话）。法则为行业通识（剪影测试/60-30-10/形状语义/视觉层级），内部知识，未逐条引证。

1. **配色结构（60-30-10）**：意图 = 层级一眼分清。世界/背景 60%（压场）、角色敌人 30%、可交互与奖励 10%（最跳）。判据：**对比分离——明度差 ≥40 或饱和度差 ≥20，择一达标**（玻璃族走明度、卡通族走饱和度；实测玻璃砖 S36/V100 在深 V29 底上靠明度分离，死卡 S≥60 会误杀好设计）。
2. **形状语义**：意图 = 不看字读懂威胁与奖励。收集物/奖励 → 圆/星形；敌人/障碍 → 尖角/不规则；地形/砖 → 矩形/圆角矩形。判据：剪影测试（第 6 条）。
3. **比例与视觉权重**：意图 = 主角是主角。尺寸层级 ≤3 档；视觉复杂度随权重递减（主角细节最多、收集物极简）；色带位置可携带信息（如高分行同色带）。判据：contact sheet 目检 + 尺寸表符合设计规范。
4. **构图规范**：意图 = 玩法区永远呼吸。背景负空间 ≥60%（玩法区上空）、焦点 ≤1；平铺件负空间 ≥30%；角色展示位三分法、焦点落上半身。判据：背景细节密度实测（玩法区显著低于装饰区，实测 0.6 vs 4.0）。
5. **产出方式决策**：意图 = 精度与质感各取所长。精度>质感 → Canvas 程序化（HUD 控件/进度条/几何收集物/HSV 色变体）；质感>精度 → AI 生图（主角衍生品/敌人/氛围背景/装饰建筑）；两者都要 → 混合（AI 底材+代码叠加）。判据：按矩阵执行，不凭空选。
6. **剪影测试（验收载体）**：意图 = 可读性终极裁判。`make_contact_sheet.py --silhouette` 全套涂黑拼图——可交互元素纯黑轮廓必须可辨；碎屑/残影无处遁形（实案：launch 帧顶边碎块被抓）。判据：剪影拼图目检全可辨、无碎块。

## 素材设计规范（平台跳跃实例；锚定 TILE 的思想通用）

> 尺寸规范数值是平台跳跃的实例；**"尺寸锚定游戏内单位、不靠 AI 自己把握大小"的思想对所有类型通用**——新类型把 TILE 换成本类型的基本单位（棋子格/平台宽），照此立规。

**尺寸一律锚定 TILE=45**（游戏内尺寸按下表绘制，不靠 AI 自己把握大小；AI 只负责画风与内容）：

| 素材 | 游戏内尺寸 | 规则 |
|---|---|---|
| 主角（两足/竖高） | 高 ≈88px（2 格） | 碰撞盒 ≈34×84，锚脚底中心 |
| 主角（四足/横宽） | 高 ≈62px（1.4 格） | 碰撞盒 34×50，锚脚底中心 |
| 收集物 | 30-36px（0.7-0.8 格） | 锚中心 + 旋转/脉动；太小看不见、太大像障碍 |
| 敌人 | 高 38-46px（0.85-1 格） | 必须比主角矮——"可踩"的视觉前提 |
| 障碍物 | 宽 ≈1 格、高 2-3 格 | 贴图比碰撞体每边大 4-6px 更自然 |
| 地形砖 | 45×45 整格 | 平铺无缝；草皮/泥土/砖同族同描边 |
| 终点建筑 | 高 ≈150px（3-4 格） | 全程最大单体素材，承担视觉终点 |
| 背景 | 3:2、地平线在下 1/3 | 低对比不抢戏；禁假平台（见"失败处理"） |

**视角统一**：角色/敌人 side view facing right（移动方向）；收集物/障碍/建筑 front view；全套只用一个风格锚点。

**风格一致性验收**：全套生成后用 `scripts/make_contact_sheet.py` 拼 contact sheet 目检——描边粗细一致、平涂上色、明暗方向统一、饱和度同区间；单件跑偏只重新生成那一件（见"失败处理"）。

**内容占比**：透明素材内容包围盒 ≥80%，裁到内容边缘（留白过多 → 游戏内视觉缩水）。

**平铺素材重复感控制**（打砖块"面具墙"实踩）：网格平铺素材（砖/地块/格子）——
- **单件去表情化**：无点状特征、无居中方向性高光（人眼对"两点一线"脸型敏感，同一砖贴 60 次变面具军团；prompt 显式写"没有任何表情和五官，没有点状图案"）；
- **色变体**：同一砖形 × 3~5 令牌色（行色带或混排，高分行同色带作分值暗示）；单件 AI 生成后 **HSV 旋色派生**其余色（脚本化一次出齐），**禁止逐色分别生成**（风格必漂移）；
- **平铺预览验收**：contact sheet 之外必须 3×3 平铺拼接目检（`make_contact_sheet.py --tile 3`）——单件好看不算数，看"重复后读起来像什么"。

**序列帧**：与静态主角同内容高度、同画风；帧间脚底对齐不漂移；循环 A→B→C→A（详见后文）。

## 主题推导策略（通用，替代查表）

不查表，从角色特征推导。阶段 A 输入角色特征表，输出主题域、槽位理由和生产方式；阶段 B 才把已批准清单转换成 prompt 或 Canvas 配方。

### 第一步：归纳主题域

把特征提取表浓缩成 1-2 个主题词（素材世界的"题材"），优先级从高到低：

1. **角色本人元素**：标志性配饰/道具/图案（心形腰包 → "甜心街头"；耳机+舞鞋 → "街舞之夜"）
2. **身份/职业**：舞者、木匠、学生、厨师（木匠 → "山野木作"）
3. **世界观场景**：城市、森林、海边、太空
4. 以上都抓不到 → 用通用默认"星星冒险"（含合理性的最低保障，不是首选）

### 第二步：槽位填充规则（每个槽位都要能说出"为什么是它"，写进 GDD）

| 槽位 | 推导规则 | 示例（甜心街头主题） |
|---|---|---|
| 收集物 | 角色喜爱/标志性的小物件，要"小而亮"，一眼想捡 | 爱心（来自心形腰包） |
| 敌人 | 与主题相冲的小怪，"凶但可爱"、比主角矮 | 捣蛋小恶魔/气鼓鼓史莱姆 |
| 障碍物 | 主题场景里的常见静态物 | 路障、消防栓 |
| 终点建筑 | 角色的"家/目标"的具象化 | livehouse / 舞台 |
| 地形砖 + 背景 | 主题场景的地表材质与远景 | 砖墙 + 城市天际线 |

反例自检：任一槽位答不上"它和角色有什么关系"→ 退回第一步重新归纳主题域，不许默默用默认项。

## 阶段 B：素材生产与验收

进入本阶段前，确认项目根已有通过设计门的 `GDD.md`。每件产物同步更新 `ASSET_LEDGER.md`：职责、目标文件、AI/代码/混合、prompt 或配方、参考图、批准证据、日期、版本和验收状态。重新生成必须更新原记录，禁止记录漂移。

`ASSET_LEDGER.md` 每件一行，字段至少包括：

| 职责 | 目标文件 | 生产方式 | prompt/Canvas 配方 | 参考图 | 批准证据 | 日期/版本 | 验收状态 |
|---|---|---|---|---|---|---|---|
| 标题页主视觉 | `cover.jpg` | AI | 完整 prompt，不写摘要 | `hero.png` | 不适用 | `YYYY-MM-DD / v1` | 待验收/通过/重做 |
| 主角 Seed | `sprites/hero/seed.png` | AI 编辑 | 角色形态 + 游戏构图简报 | 用户原图 | 用户批准日期/消息 | `YYYY-MM-DD / v1` | 待批准/通过/重做 |

### 产出形态 A：生图提示词（要调生图模型时）

**prompt 产出策略（通用，四定向写法）**——"主题词+锚点+排除词"的约束串不是提示词，是抽盲盒；合格 prompt 必须带**构图/明暗/密度/质感**四个定向。输入 = 角色图特征提取表 + 需求 + 类型槽位；产出 = 每件素材一条**中文** prompt（色值保留 HEX，跨语言精确）。骨架：

```
{物件本体+构图定向}，{视角}，{槽位用途约束}，
{风格锚点}，{色彩定向(色值)}，{线条定向}，{氛围词}，
{角色世界感引用 1-2 个元素}，单件独立主体。{槽位排除词（不要…）}
```

角色图各字段进 prompt 的位置（通用映射，不是死表）：

| 特征表字段 | 进 prompt 的位置 | 规则 |
|---|---|---|
| 画风 | 风格锚点 | 全套共用同一句（按画风速查表换写法） |
| 主色+点缀色 | 色彩定向 | 1~2 个 HEX 色值（"冰蓝 #7AC8F5 加深海军蓝 #16224A"），禁写"蓝色系"虚词 |
| 描边 | 线条定向 | 粗描边 / 细线条 / 无描边，全套一致 |
| 性格/世界观 | 氛围词 | 1 个（梦幻/街头/田园），定情绪不定内容 |
| 标志性元素 | 世界感引用 | "{元素}元素，{角色一句话}的世界观"——借世界感不复制本人（防 AI 直接画角色；唯一例外见下"角色衍生品"） |
| 形态类别 | 不进 prompt | 进设计规范（尺寸/碰撞盒），职责分离 |

槽位用途约束（类型给的硬指令，prompt 的另一半）：

- **背景**：见"背景三层方案"——滚动类构图均匀化 + 亮度锚 + `不要人物`；静态可分区（顶部压暗衬 UI/砖墙）
- **平铺砖/格**：去表情（"没有任何表情和五官，没有点状图案"）+ 无方向高光 + 正面平视 + "用于网格平铺"
- **角色衍生品**（发射手/驾驶员序列帧）：**唯一允许复制角色本人**——"以已批准 Seed 为第 1 帧，同一个{角色描述}，{动作}，单行 N 格水平序列，每格{逐格动作}"，走本文 Seed → 整条 → 归一化流水线
- **敌人/收集物**：槽位填充规则（凶但可爱/小而亮）

before/after 对照（shmup 背景实案）：
- ❌ 约束串："星空云海，kawaii 风格锚点，no platforms, no blocks…" → 随机星空壁纸 + 硬接缝 + 照片级"土"
- ✅ 四定向："竖版射击游戏滚动背景：辽阔深夜天空均匀铺满整个画面，稀疏小星与极淡星云薄雾全画面均匀分布，上下结构一致，没有地平线，没有明亮焦点。整体非常暗、低对比、低细节密度。梦幻赛璐璐动漫风格，柔光质感，冰蓝 #7AC8F5 加深海军蓝 #16224A 配色，细线条，星星元素，星光少女的世界观。不透明背景，2:3。不要人物、不要行星、不要地面、不要建筑、不要文字、不要水印。" → 均匀暗星空，mirror-wrap 后缝消失

角色特征引用的写法：不复制角色本人，只借"世界感"，如 `{星形耳坠}元素，星光少女的世界观`（英文生图环境可用 `in the world of an orange twin-tails anime girl, heart motifs, street style`）。

### 产出形态 B：Canvas 程序化绘制配方（不生图/生图失败回退时）

程序化素材不是"能画就行"，同样受设计规范约束（尺寸表/配色令牌/风格统一）。统一风格手段：**基础几何形 + 令牌色填充 + 统一粗描边（lineWidth ≈ max(2, 尺寸×0.08)）+ 圆角/高光**，靠"粗描边+平涂"的一致性掩盖细节缺失：

| 物件类别 | 几何构成配方 |
|---|---|
| 爱心/星形 | path 矢量（心=两弧+三角；星=五角 path），主色填充 + 深色描边 + 1 处高光点 |
| 圆胖敌人 | 半圆 dome + 两眼 + 怒眉，身体主色，底部椭圆接触影 |
| 柱状障碍 | 圆角矩形主体 + 顶部帽檐 + 1-2 条横向纹饰 |
| 地形砖 | 整格底色 + 顶部 1/4 高光带（草皮）或砖缝线（砖块） |
| 建筑 | 矩形主体 + 三角/拱形顶 + 门窗 2-3 个矩形，不超过 3 色 |

|---|---|---|---|---|

## 基础素材生产循环

本节用于背景、砖块、收集物、敌人和 UI 底材；**角色动作必须走后文 Seed/Strip 流水线，不能在这里逐帧批量抽卡**。

1. **准备参考**：保留用户原图；需要透明角色参考时用 `remove_bg.py`，并用 `extract_palette.py` 记录项目令牌色。不得为了方便覆盖原图。
2. **逐职责生产**：按 GDD 素材清单逐件调用当前环境可用的图片生成/编辑工具。每件只解决一个职责；可并行不同物件，不能并行生成同一角色的独立动作帧。
3. **透明策略**：工具支持可靠原生透明时直接输出 RGBA；不支持时使用与主体不冲突的纯色 chroma key，再用环境提供的去底工具处理。毛发、尾巴或半透明边缘不合格就重做，禁止把糙边带入游戏。
4. **清理与规格化**：透明小件用 `clean_sprite.py` 清水印碎块并裁内容包围盒；不透明背景按构图安全区裁切并压尺寸。不要默认裁固定百分比，先目检水印和构图位置。
5. **职责验收**：更新 `ASSET_LEDGER.md`，做 contact sheet、剪影或平铺预览，检查画风、尺寸、alpha、重复感和实际游戏构图；不合格只重做该件。

## Prompt 约束

- 风格锚点来自角色特征表，不能把 kawaii、像素或写实设成通用默认；同项目全套共用已批准锚点。
- 物件 prompt 一句话说清本体、视角、职责和构图；独立小件要求单主体，背景要求玩法区密度边界。
- 输出比例和分辨率按当前工具能力选择最接近目标的档，再在本地按 GDD 游戏内尺寸归一化；工具档位不是设计规范。
- 透明与参考图传递方式依环境适配，不能在 reference 中假定某个供应商命令或公开 URL 流程。

## 角色序列帧：Seed → 整条 → 归一化 → 验收

**输入是 GDD 模块 6 的动画决策表、生产白名单和 Sprite Contract**；本节只负责把白名单动作稳定变成游戏帧，不在生产阶段自行加减状态。用户侧输入仍只是原图和游戏需求，候选动作须先在 GDD 通过四道必要性门。

### 1. Sprite Contract 交接门

每个动作必须有：

| 字段 | 内容 |
|---|---|
| 决策证据 | M/S 档；归属、信息增量、姿态必要性、节奏收益结论；已列入生产白名单 |
| 职责/事件 | A/B/C 档；哪个游戏事件触发 |
| 动作节拍 | 逐帧语义，如持球→蓄力→投出→收势 |
| 播放契约 | 帧数、FPS、循环、中断/退出、优先级 |
| 空间契约 | 视角、游戏内尺寸、root anchor、手/球/武器等 socket |
| 物理事件帧 | 哪一帧接管/释放球、子弹、道具或命中判定 |
| 生产与回退 | 真实帧/程序化次级反馈、采样方式、加载失败表现 |

未列入生产白名单的动作不得创建 prompt、文件、加载项、状态机或测试状态。白名单内的 M 动作若被写成“静态原图 + 整体平移/缩放/染色”，设计门直接失败。

### 2. Seed Frame 批准门

1. 先生成一张**游戏内待机/起势 Seed Frame**，保留原图身份、轮廓族、配色、服饰/标志元素，但姿态和视角服务玩法。
2. 同时做一张游戏构图预览，显示角色实际尺寸、玩法区、判定点和动作节拍；透明单图不能独立代替。
3. 把 Seed + 生产白名单动作节拍给用户看一次并等待批准。批准不扩张白名单；未批准不得生成整条、分配索引或改引擎。
4. 记录 `ASSET_LEDGER.md` 的 Seed 文件、prompt、构图图、批准日期/消息和版本。同身份只换已批准色变体可不重复停顿。

### 3. 单动作整条生成

每个动作用一次编辑请求生成一条水平 strip，禁止逐帧单独生成。Prompt 必须锁定：同一角色、同朝向、同轮廓族、同配色/服饰比例、透明底、确切帧数和单行等分格，无场景/文字/海报构图。Action 段逐帧写 Contract 的动作节拍。

生产色底：环境无真透明编辑时，用不与角色冲突的纯色 chroma key，本地去底后再归一化；尾骨、毛发、半透明等复杂边缘去底失败时停下，不用糙边帧进游戏。

### 4. 构建、归一化与 socket

1. 构建带 Seed 的透明编辑画布：
   ```bash
   python3 scripts/build_sprite_edit_canvas.py --seed <seed.png> --out <action-edit-canvas.png> \
     --frames 4 --slot-width 256 --slot-height 256 --canvas-width 1024 --canvas-height 1024 \
     --resample lanczos
   ```
2. 生成整条后统一归一化：
   ```bash
   python3 scripts/normalize_sprite_strip.py --input <action-raw.png> --out-dir <sprites/action> \
     --frames 4 --frame-width 256 --frame-height 256 --anchor <seed.png> --lock-frame1 \
     --resample lanczos --socket-map <action-sockets.json>
   ```
3. **分辨率不是视觉尺度**：生成 strip 与批准 Seed 常来自不同像素分辨率。共享游戏尺度由 strip 各帧的内容包围盒决定；锁定帧 01 时，再把批准 Seed 适配到生成条第 1 帧的目标视觉框。禁止把 Seed 原始像素高直接与 strip 格子像素高取最大值，否则后续姿态会被错误缩小。
4. 采样方式按画风选：像素画用 `nearest`；绘本/动漫/扁平矢量/厚涂用 `lanczos`。不得用像素画默认毁掉插画边缘。
5. root anchor 对齐世界位置，全帧固定；交互 socket 坐标存入 `sprite-meta.json`，可逐帧移动。`socket-map` 的坐标是原始 strip 每格内的 `[x,y]`；帧 01 锁 Seed 时，其坐标以 Seed 图为准。metadata 为每帧记录实际 `source_scale`，并记录生产所用 `alpha_threshold`，供 Seed 锁定审计按同一裁框口径复现。
6. 整条生图偶尔会把相邻格的一小块耳朵/尾巴串进当前格。preview 若看到**独立连通碎块**，先确认它不是角色必要部位，再用 `--component-min-ratio 0.02` 重跑归一化；该参数只删小于最大主体 2% 的断开组件，并写入 metadata。不得用它掩盖大面积跨格或断肢，那种情况必须重生成整条。
7. 循环动作确保首尾连续；事件动作不强复制首帧，而是检查起势—物理事件帧—收势。

### 5. 预览与入引擎硬门

```bash
python3 scripts/render_sprite_preview_sheet.py --frames-dir <sprites/action> \
  --meta <sprites/action/sprite-meta.json> --out <action-preview.png> --columns 4
python3 scripts/audit_sprite_frames.py --frames-dir <sprites/action> --expected 4 \
  --meta <sprites/action/sprite-meta.json> --anchor <seed.png> --min-distinct 3
```

先放大目检 preview：身份、轮廓、服饰、姿态节拍、透明边缘和 socket 是否正确；再在实际游戏尺寸播放，检查动作可读、切换不跳、物理球/子弹真正跟随 socket 且在 Contract 指定帧释放。脚本 PASS 但放大/引擎预览难看仍判失败。

平台跳跃播放实例见 [platformer-patterns.md](platformer-patterns.md) 的“跑步序列帧动画”；其他类型按项目 GDD 的事件映射实现。

## 回退注入模式

每个素材加载后先检查、失败走代码手绘，游戏永不白屏：

```js
const boneImg = new Image(); boneImg.src = "bone.png";
const imgReady = im => im.complete && im.naturalWidth > 0;
function drawBoneShape(x, y, s) {
  if (imgReady(boneImg)) { /* drawImage 按原图宽高比绘制 */ return; }
  /* …程序化手绘回退… */
}
```

绘制时保持原图宽高比：`h = w * (im.naturalHeight / im.naturalWidth)`；角色/敌人锚定脚底中心；地形砖按 TILE×TILE 绘制。**标题页/结算页等展示位同样必须等比，禁止写死宽高硬拉伸**（竖高立绘硬拉进横宽框会直接压扁）。抠图/清理后目检内容包围盒：留白过多会让"按图高绘制"的角色视觉缩水，必须裁到内容边缘。

## 素材体积纪律

AI 生成的 PNG 单张常在 1MB 上下，全套素材很容易到 10MB 级——手机网络下加载要十几秒，加载门的等待会直接被玩家感知。约束：素材总量控制在 ~5MB 内；不透明大图（背景、地形砖）优先导出时压尺寸/转 JPG 或 WebP；透明小件保持 PNG。单张纹理单边压到 ≤1024（AI 常出 1536+，3D 里每张大图都是数 MB 显存，手机会丢 WebGL 上下文整屏变底色）。

## 失败处理

- 生图服务暂时不可用：按工具建议退避重试；高分辨率反复失败就先降档，不并行连抽。
- 背景画出"假平台/假砖块"误导玩家：重新生成，prompt 加 no platforms / no blocks。
- 抠图后为空：容差调小；背景太复杂就换 AI 生成主角（描述尽量贴合原图特征）。
- 单个素材风格跑偏：只重新生成那一个，prompt 里补一句与其他素材一致的具体特征。
