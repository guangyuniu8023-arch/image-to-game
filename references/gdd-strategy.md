# 通用 GDD 设计策略

所有 0-1 游戏都使用本策略。它规定**必须闭合哪些设计决策及其输入输出**，不规定项目 GDD 的标题、表格、篇幅或书写顺序。简单游戏可以合并模块；不适用项写明理由，不得无声跳过。

## 文件职责与项目产物

- 本文件负责通用设计覆盖面。
- 已有类型包提供验证过的类型基线和差异，不能代替项目设计。
- [new-type.md](new-type.md) 只在没有类型包时补充考据、红线推导和类型蒸馏流程。
- [assets.md](assets.md) 与 [ui-kit.md](ui-kit.md) 的“阶段 A”提供设计判据；“阶段 B”只执行已通过设计门的决策。
- 每个项目在根目录保存 `GDD.md`。它才是该游戏真正的 GDD。
- 每个项目按 [visual-framing.md](visual-framing.md) 从本次玩法需求另生成 `VISUAL_CONTRACT.json`。GDD 只记录镜头/构图的设计理由；机器合同记录实际对象、状态和断言，二者都不得变成类型选择表。
- 素材 prompt、Canvas 配方、生成日期和文件状态写入项目 `ASSET_LEDGER.md`，不混入 GDD 或类型包。

已有类型项目允许继承类型包结论，但 `GDD.md` 必须写明“继承什么、调整什么、为什么”。模板换皮可以写成差异 GDD，不必重抄不变的玩法基线。

## 输入与总输出

**输入**：用户需求、用户图片事实特征、平台/时长/受众等明确约束、选中的类型包或新类型考据结果。

**总输出**：一组可执行、可追溯的设计决策，覆盖下列八模块，并在模块 8 通过玩法—角色—素材—Sprite—HUD—视觉策略—验证一致性门。

## 八个设计模块

### 1. 核心体验、循环与输入

- 输入：用户需求、游戏类型、图片事实特征和明确约束。
- 写清玩家扮演谁、每 10 秒做什么、承受什么压力、怎样获得进展。
- 定义核心循环、主要动作、单局目标、WIN/LOSE 和输入维度；按 [controls.md](controls.md) 映射语义输入。
- 输出：核心体验、玩家动作、循环、输入模式和胜负条件。

### 2. 机制、参数与可行性红线

- 输入：模块 1、类型包或新类型考据结果。
- 给机制参数写明来源与理由；影响可玩性的硬数字必须由公式、模拟或机器人实证收口。
- 每个新类型至少推导 3 条红线，使用“风险 → 算术/不变量 → 代码约束 → 验证证据”。已有类型继承红线时记录来源，改参数时重推牵连项。
- 输出：规则、参数基线、来源、红线和调参牵连项。

### 3. 关卡、内容与进度

- 输入：玩家动作、胜负条件、机制能力边界和红线。
- 关卡型设计教学—强化—组合—高潮；无关卡型设计时间/步数/目标/密度/解锁等参数进度。
- 每个敌人、障碍和道具必须改变玩家决策；只增加数量而不改变决策的内容合并或删除。
- 输出：单局内容结构、难度曲线和可由机器人检查的完整性条件。

### 4. 角色职责、主题与视觉方向

- 输入：原图事实特征、玩家身份/动作和玩法实体。
- 参考 [assets.md](assets.md) 阶段 A，选择 A 操作主体、B 主体驱动者或 C 看板娘；角色必须承担操作、驱动或解说职责。
- 从标志元素、身份和世界观推导主题映射；每个主要实体说明为什么属于这个角色的世界。
- 定义画风、描边、主色/点缀色、形状语义、对比和密度方向。
- 同时分析玩家在关键状态判断什么、必须共同看见哪些实体/路径、哪些输入依赖稳定参考系；把事实交给 visual-framing.md 推导组合式策略，不根据游戏类型名称选择镜头。
- 输出：角色职责、主题映射、视觉设计简报和视觉决策事实。

### 5. 素材系统设计

- 输入：模块 1~4 的玩法实体、反馈事件、内容结构、角色职责和视觉简报。
- 从玩法和信息职责反推素材，不从某个类型的固定清单照抄。
- 每项素材闭合：职责、状态/变体、视角、游戏内尺寸锚点、透明像素框或几何测量方式、绘制层、与碰撞盒关系、AI/代码/混合生产方式、加载失败回退。
- 背景和装饰也要声明密度边界，避免产生假平台、假障碍或假按钮。
- 输出：可交给 assets.md 阶段 B 的素材生产清单。

### 6. Sprite 与动画状态设计

- 输入：模块 1 的玩家动词与事件、模块 4 的角色职责、模块 5 的角色素材/尺寸/锚点，以及 HUD、玩法物体和特效已能表达的信息。
- 先产出**动画决策表**，每个候选事件记录：角色与事件的关系、玩家必须读懂的新信息、现有表达渠道、是否依赖身体轮廓/root/socket/碰撞/事件帧、播放对输入和节奏的影响、档位、最终表现和是否生产。状态名不构成默认清单。
- 对每个候选执行四道必要性门：**归属**（角色是直接执行者或物理承受者）、**信息增量**（现有物体/HUD/特效不能充分表达）、**姿态必要性**（需要新轮廓、空间关系、交互 socket、碰撞或明确事件帧）、**节奏收益**（可读性收益高于播放延迟、资源加载和验证成本）。任一项不成立时，默认不生产新动作，改用无额外表现、复用已有姿态、UI 或程序化次级反馈。
- 用 **M / S / P** 管理动画预算：M 机制动作通过四门后必须生产与媒介匹配的真实姿态变化（2D 为不同姿态帧，3D 为骨骼或几何关节姿态）；S 状态说明默认先用更低成本表现，只在没有它会产生状态歧义时升级；P 装饰反馈不进入初始动画生产。角色职责 A/B/C 只决定“归属”的判断方式，不自动增加任何动作。
- 只把 M 和必要的 S 写入**生产白名单**与 **Sprite Contract**。Contract 中每个动作记录触发事件、四门结论、语义节拍、帧数与顺序、FPS、循环、中断/退出条件、朝向、root anchor、交互 socket、物理事件帧和回退。未入选项只在 GDD 记录简短排除理由，不创建文件、加载项、状态机或测试状态。
- root anchor 维持角色在世界中稳定；球/手/武器等交互 socket 可逐帧移动。物理球、子弹或道具必须在 Contract 指定的事件帧从 socket 接管/释放，不得从角色外随意跳出。
- 输出：动画决策表、生产白名单、Sprite Contract、Seed Frame 构图简报和引擎事件映射。

### 7. HUD/UI 信息与视觉设计

- 输入：决策变量、胜负/资源状态、布局约束和视觉简报。
- 先决定玩家做当前决策必须知道什么，再决定如何显示；区分常驻信息、情境反馈和角落操作。
- 参考 [ui-kit.md](ui-kit.md) 阶段 A，定义信息优先级、屏幕分区、组件/状态、主题 icon、标题/结算结构及 UI 线条、质感、形状、密度。
- 输出：HUD 信息架构、布局方向、组件状态清单和 UI 四属性。

### 8. 架构、验证与一致性门

- 输入：模块 1~7 的全部决策。
- 定义状态机、自然启动状态、主循环骨架、数据表示、资源加载门、机器人操作方式、WIN/LOSE 判定、死局处理和关键截图场景。标题/封面必须是自然启动后可达的真实状态，不能只在测试入口里强制显示。核心状态转换统一由 `update(dt)` 或等价可控时钟推进，不依赖真实 `setTimeout`；逐状态写最大模拟帧数，并为角色旋转、缩放、位置等渲染状态写有限值/范围不变量。
- 按 visual-framing.md 把模块 1~7 的视觉决策事实推导成 `VISUAL_CONTRACT.json`：必须共同可见对象、参考系敏感状态、重新取景事件、组合式相机行为和运行时 cases。不得要求用户选镜头，或从类型名复制固定策略。
- 在完整素材生产前建立最小真实构图骨架，暴露只读 `window.__game.visualAudit.snapshot` 和确定性 `window.__game.visualAudit.runCase`；Seed 构图预览和正式审计必须复用同一舞台、投影、相机与 HUD 安全区。重新取景 case 必须记录触发前 probe，并从世界位移与投影推导最小 target 位移、镜头实际位移和必须变化的屏幕轴；主实体 probe 必须报告真实 `render_source`。
- 验证方案必须按类型路由；新类型要建立项目适配机器人或确定性规则测试，不得套用平台跳跃脚本。另生成 `GAMEPLAY_CONTRACT.json`，把成功、每个 GDD 失败和边界规则的全部实际侧写成独立 case；单侧规则不编造第二侧，多侧规则不得缩减。机器人输出哈希绑定的 `evidence/gameplay-audit.json`，不能用“成功路径跑很多次”代替失败与边界验证。
- 生产前逐项检查：玩法实体有视觉表达与回退；动画决策表已过四道门；加载、状态机和测试中的每个动作都在生产白名单中；M 动作有与媒介匹配的真实姿态变化和事件时间线；root/socket 关系明确；关键数值进入 HUD 或说明无需显示；素材尺寸与碰撞/布局一致；HUD 不遮挡玩法；视觉合同能追溯到本次需求；不存在无职责素材或冗余动作。
- 输出：实现架构、自动验证口径、`VISUAL_CONTRACT.json`、`GAMEPLAY_CONTRACT.json`、截图状态清单和一致性结论。任何一项不闭合，退回对应模块。

## 设计门通过后的交接

- 给 assets.md 阶段 B：视觉简报、素材生产清单、动画决策表、生产白名单、Sprite Contract 和 Seed Frame 构图简报。
- 给 ui-kit.md 阶段 B：HUD 信息架构、布局方向、组件状态和 UI 四属性。
- 给实现：状态机、语义输入、数据结构、锚点/碰撞关系和回退要求。
- 给验证：机器人入口、成功/失败/边界 case、各规则由 GDD 声明的完整单侧/多侧覆盖、内容完整性断言、每状态最大模拟帧数、渲染状态不变量、关键截图状态和布局要求。
- 给视觉运行时门：本次需求推导出的合同、确定性 cases、真实可见像素/几何 probe、Seed 基线和正式版本比较条件。

生产中发现缺状态、动作不可读、尺寸/socket 冲突、HUD 遮挡或无职责素材时，必须回写 `GDD.md` 并重新通过设计门；不得在生产阶段静默改写设计。
